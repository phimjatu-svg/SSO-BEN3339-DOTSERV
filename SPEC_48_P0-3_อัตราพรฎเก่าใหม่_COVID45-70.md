## 1. Overview

แยกงานออกเป็น 2 epic อิสระต่อกัน:

- **EPIC A — อัตราตาม พรฎ เก่า/ใหม่ (คร่อมขอบกฎ → คำนวณ 2 อัตรา):** รองรับคำขอเดียวที่ช่วงสิทธิคร่อม "วันมีผล/วันสิ้นสุด" ของกฎกระทรวง โดยแบ่งช่วงจ่ายเป็นหลาย sub-period คิดคนละอัตรา/คนละระยะเวลา และแสดงผลใน UI
- **EPIC B — ช่วงโควิด 45/70 (เปิด claim ใหม่ ไม่คร่อม):** เพิ่มอัตราโควิด 45% (ลาออก) / 70% (เลิกจ้าง) เป็นประเภทเคสที่จัดการแบบ claim เดียวอัตราเดียว ไม่ทำตรรกะคร่อมขอบกฎภายในเครื่องมือ

> **หมายเหตุความขัดแย้ง:** เอกสารต้นทางแสดงว่าคำขอช่วงโควิด "มีการคร่อมจริง" (ดู §7 ตัวอย่าง 1–4) EPIC B เป็นการตัดสินใจลดความซับซ้อนเชิง implementation ไม่ใช่กติกาตามกฎหมายเป๊ะ — ต้องยืนยันขอบเขต (Q2, §9)

## 2. Scope

**In scope**
- ตรรกะแบ่ง sub-period ตามวันขอบกฎ (boundary date) + คิดอัตรา/ระยะเวลาต่อช่วง (EPIC A)
- เพิ่มอัตรา/เพดานวันโหมดโควิด (EPIC B)
- การแสดงผล breakdown 2 อัตราใน UI (timeline + ตารางงวด)
- Golden test 4 เคสจากเอกสาร (§8)

**Out of scope**
- การเปลี่ยนสูตรค่าจ้างรายวัน ม.57 (คงเดิม)
- การเปลี่ยนกฎปัดเศษ 0.05 (คงเดิม)
- ตรรกะความซ้ำซ้อน/เรียกเงินคืน (อยู่ใน ticket อื่น)

## 3. Background — กติกาที่ใช้คำนวณ (อ้างอิงเอกสาร)

### 3.1 อัตรา & เพดานวัน ต่อช่วงกฎ

| ช่วงกฎ | เลิกจ้าง (R3) | ลาออก/สิ้นสุดสัญญา (R1/R2) |
|---|---|---|
| กฎ 2547 (ก่อน 1 มี.ค.63 และ หลัง 28 ก.พ.65) | 50% · ≤180 วัน | 30% · ≤90 วัน |
| โควิด (1 มี.ค.63 – 28 ก.พ.65) | **70% · ≤200 วัน** | **45% · ≤90 วัน** |
| กฎ ฉบับที่ 2 พ.ศ.2568 (28 มิ.ย.68 เป็นต้นไป) | 60% · ≤180 วัน | 30% · ≤90 วัน |

### 3.2 โควตารวม/ปีปฏิทิน

- ช่วงโควิด: มีเหตุเลิกจ้างปน → รวม ≤ 200 วัน · ลาออกล้วน → รวม ≤ 90 วัน
- ช่วงปกติ (2547/2568): มีเลิกจ้างปน → ≤ 180 · ลาออกล้วน → ≤ 90

### 3.3 กติกาคร่อม (transition) — 2 ทิศ
1. **เก่า→ใหม่ (ขอบเข้า 1 มี.ค.63):** ผู้รับสิทธิตามกฎเดิมอยู่ก่อน + ยังไม่ครบ → ส่วนที่เหลือปรับเป็นอัตรา/ระยะเวลาที่สูงขึ้นตามกฎใหม่ เฉพาะวันที่เหลือ
2. **ใหม่→เก่า (ขอบออก 28 ก.พ.65):** ผู้ยังมีสิทธิหลังกฎหมดอายุ → ส่วนที่เหลือกลับไปใช้อัตรา/ระยะเวลากฎเดิม

## 4. โมเดลหลัก — ปิดรับแจ้ง #1 + เปิดรับแจ้ง #2 (Close & Reopen)

> **ตัดสินใจแล้ว (ผู้ใช้ยืนยัน):** กรณีคร่อมสิทธิ ระบบ **ไม่** แตก sub-period ภายในรับแจ้งเดียว แต่ใช้วิธี **ปิดรับแจ้ง #1 ที่ขอบกฎ แล้วเปิดรับแจ้ง #2** (อัตรา regime ใหม่) ต่อเนื่องกัน — วิธีนี้ใช้กับ **ทุก regime** (รวมโควิด) จึงรวม EPIC A เดิม + EPIC B เดิม เป็นกลไกเดียว และ **ปิดคำถาม Q1/Q2 เดิม**

### 4.1 Functional Requirements (EPIC A — Close & Reopen)

**A-1** เก็บ "ตารางวันขอบกฎ" (rate regime table) เป็น config ไม่ hardcode — แต่ละ regime มี: `effectiveFrom`, `effectiveTo`, `rateLayoff`, `rateResign`, `capLayoff`, `capResign`, `wageCeiling`

**A-2 (ตรวจจับคร่อม)** เมื่อช่วงสิทธิของรับแจ้ง (วันเกิดสิทธิ → วันครบสิทธิ) คาบเกี่ยว ≥2 regime → **ระบบเสนออัตโนมัติ** ให้ปิด+เปิดรับแจ้ง (ไม่ทำเงียบ ๆ) แล้ว **เจ้าหน้าที่กดยืนยัน** ก่อนผูกผล

**A-3 (ปิดรับแจ้ง #1)** รับแจ้ง #1 = วันเกิดสิทธิเดิม → **(วันขอบกฎ − 1)** ; ปิดสถานะที่วันนั้น ใช้อัตรา/เพดานของ regime เดิม

**A-4 (เปิดรับแจ้ง #2)** รับแจ้ง #2 = **วันขอบกฎ → วันครบสิทธิ** ; ใช้อัตรา/เพดานของ regime ใหม่
- **ไม่มี waiting 8 วันใหม่** — วันเกิดสิทธิ #2 = วันขอบกฎ จ่ายต่อเนื่องทันที
- **นับจำนวนวัน + โควตา/ปีปฏิทิน ต่อจากรับแจ้ง #1** (สะสมข้ามรับแจ้ง)
- ผูกความสัมพันธ์ #1↔#2 (link/parent ref) เพื่อ trace

**A-5 (จำนวนวันสิทธิรวม)** กำหนดโดย regime ที่คุม **"ส่วนที่เหลือ"** ของสิทธิ ไม่ใช่ regime ตอนเกิดสิทธิ
- *ที่มา:* เคส 2 ได้ 200 วัน (โควิดคุมส่วนที่เหลือ) แต่เคส 4 ได้ 180 วัน (กฎ 2547 คุมส่วนที่เหลือหลัง 28 ก.พ.65) — ดู §8
- clip จำนวนวันด้วยเพดาน regime นั้น และด้วยโควตา/ปีปฏิทิน (§3.2)

**A-6 (คำนวณเงิน/รับแจ้ง)** แต่ละรับแจ้ง: `เงินต่อวัน = floor05( ค่าจ้างรายวัน(ม.57) × rate )` แล้ว `เงินรับแจ้ง = floor05( เงินต่อวัน × จำนวนวัน )` (กฎปัดลง 0.05 เดิม — ปัดเป็นสตางค์ก่อน floor)

**A-7 (ไม่ซ้ำซ้อน)** รับแจ้ง #1 และ #2 **ต่อเนื่องกัน ไม่ทับซ้อนวัน** → ตรรกะตรวจ "ความซ้ำซ้อน" **ต้องไม่ flag** คู่นี้ (สอดคล้องกฎเดิม: ตัดสินซ้ำซ้อนจากช่วงจ่ายจริงที่ทับกัน) — ต้องมี test กันfalse-positive

**A-8 (แสดงผล)** Timeline + ตารางงวด แสดง 2 รับแจ้งเป็นบล็อกต่อเนื่อง ระบายสีแยกตาม regime/อัตรา + ป้าย "อัตราเดิม/ใหม่" + tooltip วันขอบกฎ และยอดรวมข้ามรับแจ้ง

### 4.2 อัตราโควิด (เดิม EPIC B — เป็น regime หนึ่งในตาราง A-1)

**B-1** เพิ่ม regime โควิดในตาราง: `ลาออก-COVID = 45% (cap 90)`, `เลิกจ้าง-COVID = 70% (cap 200)` ผลบังคับ 1 มี.ค.63 – 28 ก.พ.65

**B-2** เมื่อรับแจ้งคร่อมขอบโควิด → ใช้กลไก Close & Reopen เดียวกัน (A-2…A-8) ไม่มีตรรกะแยกต่างหาก

**B-3** โควตา/ปีปฏิทินช่วงโควิด: เลิกจ้างปน → 200, ลาออกล้วน → 90 (มาจาก `capLayoff/capResign` ของ regime โควิดใน config — ไม่ hardcode)

## 6. Data Model

เพิ่ม/แก้ field (ชื่อ field รอ map กับ schema จริง):

โมเดล Close & Reopen → แต่ละรับแจ้งเป็น record เดียว อัตราเดียว ; การคร่อม = 2 record ที่ link กัน (ไม่มี array sub-period ภายใน record)

ระดับรับแจ้ง (claim record):

| Field | Type | คำอธิบาย |
|---|---|---|
| `rateRegimeId` | enum/ref | regime ของรับแจ้งนี้ (2547 / covid2563 / krq2_2568) — ลุคอัพอัตรา/เพดาน |
| `benefitStart` | date | วันเกิดสิทธิของรับแจ้งนี้ (#2 = วันขอบกฎ, ไม่มี waiting ใหม่) |
| `benefitEnd` | date | วันปิด/วันครบของรับแจ้งนี้ |
| `daysUsedBefore` | int | วันสิทธิที่ใช้ไปแล้วก่อนหน้า (สะสมจากรับแจ้ง #1 → ใช้นับต่อ/คุมเพดาน) |
| `linkedClaimId` | ref/null | ผูก #1↔#2 (continuation) — ใช้ trace + กันตรวจซ้ำซ้อน |
| `closeReason` | enum/null | เหตุปิด เช่น `regime_boundary` (ปิดเพราะคร่อมขอบกฎ) |

ระดับ config (ส่วนกลาง):

| Field | Type | คำอธิบาย |
|---|---|---|
| `rateRegimes[]` | config | `{regimeId, isDefault, effectiveFrom, effectiveTo, rateLayoff, rateResign, capLayoff, capResign, wageCeiling}` |

> regime table เป็น single source of truth (A-1) — เพิ่ม regime ใหม่ (เช่นเพดาน 17,500 วันที่ 1 ม.ค.69) ทำได้โดยไม่แก้โค้ดคำนวณ ; ตรรกะ "ตรวจคร่อม → เสนอปิด+เปิด" อ่านจากตารางนี้
>
> **Master / ค่าตั้งต้น (`isDefault=true`):** กฎกระทรวง พ.ศ.2547 เป็น regime ค่าตั้งต้น (ไม่มีวันสิ้นสุด) — ช่วงเวลาใดที่ไม่มีประกาศเฉพาะครอบคลุม ระบบใช้ master นี้คำนวณอัตโนมัติ จึง **ไม่มีแนวคิด "ช่องว่าง"** (gap = ใช้ master) ; ระบบต้องมี master active เสมอ 1 รายการ ; การตรวจคร่อม (A-2) นับขอบ master↔regime เฉพาะด้วย

## 8. Acceptance Criteria & Golden Tests

ใช้ 4 เคสจาก §7 เป็น automated test (ค่าจ้างรายวัน = D; โมเดล Close & Reopen → ตรวจ "รับแจ้ง #1/#2 + จำนวนวัน + อัตรา + วันปิด/ครบ")

**AC-A1 (เคส1, ลาออก คร่อมเข้า):** รับแจ้ง#1 = 8–29 ก.พ.63 (22 วัน @30%, ปิด 29 ก.พ.); รับแจ้ง#2 = 1 มี.ค.–7 พ.ค.63 (68 วัน @45%); รวม 90 วัน

**AC-A2 (เคส2, เลิกจ้าง คร่อมเข้า):** #1 = 22 วัน @50%; #2 = 178 วัน @70%; รวม **200 วัน** (cap regime โควิด), ครบ 25 ส.ค.63

**AC-A3 (เคส3, ลาออก คร่อมออก):** #1 = 8–28 ก.พ.65 (21 วัน @45%, ปิด 28 ก.พ.); #2 = 1 มี.ค.–8 พ.ค.65 (69 วัน @30%); รวม 90 วัน

**AC-A4 (เคส4, เลิกจ้าง คร่อมออก):** #1 = 21 วัน @70%; #2 = 159 วัน @50%; รวม **180 วัน** (cap regime 2547), ครบ 6 ส.ค.65

**AC-A5:** วันเกิดสิทธิ #2 = วันขอบกฎ (ไม่มี waiting); ผลรวมวัน #1+#2 = จำนวนวันสิทธิรวม; เงินรวม = เงิน#1 + เงิน#2

**AC-A6 (ไม่ซ้ำซ้อน):** รับแจ้ง #1↔#2 ที่เกิดจากการคร่อม **ต้องไม่** ถูก flag เป็นความซ้ำซ้อน (ต่อเนื่อง ไม่ทับวัน) — ดู A-7

**AC-A7 (control ไม่คร่อม):** รับแจ้งที่ช่วงสิทธิอยู่ใน regime เดียว → ระบบ **ไม่เสนอ** ปิด+เปิด; เป็นรับแจ้งเดียว อัตราเดียว

**AC-rounding:** ทุกค่าเงินผ่านกฎปัดลง 0.05 (เช็ค 583.30 → ×30% = 174.95 → ×36 = 6,298.20)

> **Edge cases:** เกิดสิทธิวันสุดท้ายก่อนขอบกฎพอดี; ช่วงสิทธิจบก่อนขอบกฎ (ไม่เสนอปิด+เปิด); ปีอธิกสุรทิน (ก.พ.63 = 29 วัน → #1 = 22 วัน); ช่วงคร่อม >2 regime (เสนอปิด+เปิดหลายทอด)
